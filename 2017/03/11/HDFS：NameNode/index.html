HDFS：NameNode

负责管理命名空间，维护HDFS的目录树以及文件的元数据信息，包括了两个关键的表：

  1. 文件到数据块的映射关系（Namespace）：存放在内存和磁盘上
  2. 数据块到DataNode的映射关系：每次启动NameNode重新建立

提供了三类接口：

  1. org.apache.hadoop.hdfs.protocol.ClientProtocol：客户端用来请求HDFS服务
  2. org.apache.hadoop.hdfs.server.protocol.DatanodeProtocol：向DataNode存放数据块
  3. org.apache.hadoop.hdfs.server.protocol.NamenodeProtocol：rebalancing进程和Secondary NameNode用来获取NameNode的状态

启动过程如下：

FSNamesystem

HDFS实际上是通过FSNamesystem进行管理 ，主要的作用是：

  1. 建立BlockManager, DatanodeManager, DelegationTokens, LeaseManager
  2. 用于处理修改和检查Namespace的RPC请求
  3. 有关数据块信息的report会交给BlockManager处理
  4. 对文件系统进行修改的操作交给FSDirectory处理
  5. log的相关操作交给FSEditLog去完成

维护了五个主要的数据结构：

  1. 有效的filename到blocklist的映射
  2. 有效的block集合
  3. 每个block对应的DataNode列表，存储在内存中，动态的根据DataNode发来的report进行重建
  4. DataNode与其上的block的映射
  5. 采用LRU cache算法来更新DataNode

FSImage

FSImage主要用来进行检查点（根据dfs.namenode.checkpoint.period和dfs.namenode.checkpoint.txns决定）操作以及对Namespace的编辑和日志记录，通常在启动时会被先加载到内存中然后，再执行FSEditLog中记录的操作，从而保证内存中的元数据和实际的元数据同步。

Non-HA模式下，Secondary NameNode定期通过HTTP的方式从NameNode上获取FSEditLog，然后合并到FSImage，并将FSImage发回给NameNode用以替换NameNode上保存的FSImage

HA模式下，通过共享系统NFS或者bookeeper将数据写入共享存储系统（包含多个JournalNode），standby对其监听并将同步改动（FSEditLog）合并到FSImage

FSDirectory

FSDirectory和FSNamesystem都用来管理和维护Namespace的状态。FSDirectory是位于内存的一种数据结构，FSDirectory的所有操作都发生在内存。而FSNamesystem主要是用来将操作持久化到磁盘

FSEditLog

用于记录日常对Namespace改动的操作，在对文件进行修改时，只有修改成功后，才会写入log，当log写入成功后再更新内存中的元数据信息。

Hadoop中EditLog拥有以下五种状态：

private enum State {
UNINITIALIZED,
BETWEEN_LOG_SEGMENTS,
IN_SEGMENT,
OPEN_FOR_READING,
CLOSED;
}

在Non-HA模式下，EditLog开始处于UNITIALIZED的状态，一旦完成初始化，EditLog的状态变为了IN_SEGMENT的状态并可以进行修改。在对EditLog进行roll或者保存Namespace时，进入BETWEEN_LOG_SEGMENTS的状态，表示之前的segment已经关闭，但是新的还没有开始

在HA模式下，EditLog开始同样处于UNITIALIZED的状态，完成初始化以后，处于standby状态的NameNode会一直保持OPEN_FOR_READING的状态。调用某命令将NameNode转化为active的过程中，状态先会变为CLOSED，然后变为BETWEEN_LOG_SEGMENTS，在开始写入EditLog时，状态变为IN_SEGMENT ，之后的过程和Non-HA模式下一样。

INode

Inode主要维护文件ID，权限，所属用户，组，属性，以及修改的相关信息等
